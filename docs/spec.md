# Specification: Workout Challenge Tracker

## 1. Overview (Обзор)

### Суть приложения
Приложение для спортивной мотивации, позволяющее пользователям создавать индивидуальные челленджи (например, "3000 махов гирей за 30 дней"), регистрировать свои тренировки, отслеживать прогресс, укладываться в дедлайны и получать достижения (ачивки).

### Цели
- Повышение спортивной мотивации через визуализацию прогресса.
- Гибкая настройка целей (упражнения, количество, сроки).
- Геймификация тренировочного процесса.

### Стек технологий и эволюция проекта
Проект разрабатывается итеративно с учетом следующих этапов:
1. **MVP (Локальный запуск):** 
   * Single-user режим, но **архитектурно закладывается multi-user** (во все таблицы сразу добавляется `user_id`, в MVP хардкодится как `default_user_1`, чтобы избежать переписывания БД в будущем).
   * **Frontend:** SPA на Vanilla HTML/JS/CSS.
   * **Backend:** Легковесный API сервер на Go с базой данных (PostgreSQL в Docker).
   * Взаимодействие через REST API.
2. **Интеграция с Telegram:** Обертка фронтенда в Telegram Mini App с авторизацией через Telegram WebApp API (здесь хардкод `user_id` заменится на реальный Telegram ID).

## 2. Architecture & Data Model (Архитектура и Модель данных)

### Сущности и поля БД (Схема данных)

#### Упражнение (Exercise)
- `id` (UUID / Integer) — Уникальный идентификатор.
- `user_id` (String / UUID) — Владелец (NULL для системных, заполняется для кастомных).
- `name` (String) — Название упражнения.
- `is_custom` (Boolean) — Создано ли пользователем вручную (true) или системой (false).

#### Челлендж (Challenge)
- `id` (UUID / Integer) — Уникальный идентификатор.
- `user_id` (String / UUID) — Владелец челленджа.
- `name` (String) — Название челленджа.
- `exercise_id` (ForeignKey -> Exercise) — Ссылка на упражнение.
- `target_value` (Integer) — Целевое количество повторений.
- `start_date` (Date) — Дата начала челленджа.
- `end_date` (Date) — Дата окончания (дедлайн).
- `status` (Enum: 'active', 'completed', 'failed') — Текущий статус. Рассчитывается динамически (если `end_date` прошел, а прогресс < 100%, статус меняется на `failed` при запросе или фоновым джобом).

#### Тренировка / Запись выполнения (Workout)
- `id` (UUID / Integer) — Уникальный идентификатор.
- `user_id` (String / UUID) — Владелец записи.
- `challenge_id` (ForeignKey -> Challenge) — Ссылка на челлендж.
- `workout_date` (Date) — Дата проведения тренировки.
- `value` (Integer) — Выполненное количество повторений.
- `created_at` (Timestamp) — Время внесения записи.

## 3. UI Requirements & App Logic (Требования к UI и логике)

### Ключевые экраны и элементы
1. **Dashboard (Главная страница):**
   * Список активных челленджей.
   * Кнопка «Создать новый челлендж».
   * Секция с полученными достижениями (ачивками).
2. **Страница челленджа:**
   * Прогресс-бар (процент выполнения цели от текущего прогресса).
   * График прогресса (накопительный итог).
   * Таймер обратного отсчета (оставшееся время до `end_date`).
   * Кнопка «Добавить тренировку» (вызывает модальное окно).
   * История тренировок (список внесенных записей по датам).
3. **Форма создания челленджа:**
   * Поля: Название, Выбор упражнения (из списка с опцией «добавить свое»), Целевое количество, Даты старта и окончания.

### Система достижений (Ачивки)
Ачивки рассчитываются динамически на бэкенде:
1. **«Первый шаг»** — внесена первая тренировка.
2. **«Экватор»** — прогресс по челленджу >= 50%.
3. **«Герой»** — прогресс >= 100% до наступления `end_date`.
4. **«Стабильность»** — тренировки вносились 3 дня подряд.

## 4. API Contracts (Спецификация API)

Все запросы используют формат JSON. Префикс API: `/api`.
*(Для интеграции с Telegram фронтенд передает заголовок `Authorization: Bearer <initData>`. Заголовок `X-User-Id: default_user_1` сохранен как fallback для локальной разработки и тестов)*

### Упражнения
* `GET /api/exercises` — получить список упражнений.
* `POST /api/exercises` — добавить кастомное упражнение. `{ "name": "string" }`

### Челленджи
* `GET /api/challenges` — получить список челленджей (с пересчитанными статусами и суммарным прогрессом).
* `GET /api/challenges/:id` — детальная информация, включая список тренировок.
* `POST /api/challenges` — создать челлендж. `{ "name": "string", "exercise_id": number, "target_value": number, "start_date": "YYYY-MM-DD", "end_date": "YYYY-MM-DD" }`
* `DELETE /api/challenges/:id` — удалить челлендж. Все тренировки удаляются каскадно (ON DELETE CASCADE). Ответ: `{ "success": true }`.

### Тренировки (Workouts)
* `POST /api/challenges/:id/workouts` — добавить тренировку. 
  * Тело: `{ "workout_date": "YYYY-MM-DD", "value": number }`
  * **Ответ (важно для UX):** `{ "success": true, "workout": {...}, "unlocked_achievements": ["equator"] }`. Фронтенд использует массив `unlocked_achievements` для показа поздравительных поп-апов.
* `DELETE /api/workouts/:id` — удалить запись. Бэкенд автоматически производит каскадный пересчет: если удаление опускает прогресс ниже 100%, статус челленджа меняется с `completed` обратно на `active`.

## 5. User Stories & Acceptance Criteria

### US-1: Создание упражнения
**Как** пользователь, **я хочу** добавлять кастомные упражнения, **чтобы** использовать их в челленджах.
* **AC-1:** *Given* пользователь на форме создания, *When* он выбирает «Добавить свое», *Then* появляется текстовое поле.
* **AC-2:** *Given* введено уникальное название, *When* форма отправляется, *Then* упражнение сохраняется с флагом `is_custom: true` и становится доступным для выбора.
* **AC-3:** *Given* пустое название, *When* форма отправляется, *Then* бэкенд возвращает ошибку `400 Bad Request`.

### US-2: Создание челленджа
**Как** пользователь, **я хочу** поставить перед собой спортивную цель с дедлайном, **чтобы** отслеживать путь.
* **AC-1:** *Given* форма заполнена корректно, *When* пользователь нажимает «Сохранить», *Then* челлендж сохраняется со статусом `active`.
* **AC-2:** *Given* пользователь выбирает `end_date` раньше, чем `start_date`, *When* он пытается сохранить, *Then* фронтенд блокирует кнопку и показывает ошибку, а при обходе UI бэкенд возвращает `400`.

### US-3: Логирование тренировки и Ачивки
**Как** пользователь, **я хочу** вносить результаты тренировок и получать награды, **чтобы** видеть прогресс.
* **AC-1:** *Given* валидные данные, *When* тренировка добавлена, *Then* прогресс-бар мгновенно пересчитывается без перезагрузки страницы.
* **AC-2:** *Given* тренировка довела прогресс до 50%, *When* API возвращает успешный ответ, *Then* ответ содержит `"unlocked_achievements": ["equator"]`, и фронтенд показывает модальное окно с поздравлением.
* **AC-3:** *Given* отрицательное число повторений, *When* отправляется запрос, *Then* возвращается ошибка `400`.

### US-4: Удаление ошибочной тренировки
**Как** пользователь, **я хочу** удалять случайные логи, **чтобы** статистика была точной.
* **AC-1:** *Given* челлендж в статусе `completed`, *When* пользователь удаляет последнюю тренировку (опуская прогресс ниже 100%), *Then* статус челленджа возвращается в `active`.

### US-5: Просмотр Дашборда (Главного экрана)
**Как** пользователь, **я хочу** видеть список моих челленджей и полученные награды на главном экране, **чтобы** быстро оценивать свой статус.
* **AC-1:** *Given* пользователь открыл приложение, *When* данные загрузились, *Then* отображается список активных челленджей (с прогресс-баром и оставшимся временем) и секция полученных ачивок (цветные иконки для открытых, серые — для закрытых).
* **AC-2:** *Given* у пользователя нет активных челленджей, *When* открывается главный экран, *Then* показывается заглушка «У вас нет активных челленджей» и кнопка «Создать первый челлендж».

### US-6: Просмотр деталей челленджа
**Как** пользователь, **я хочу** видеть подробную информацию о конкретном челлендже, **чтобы** изучать динамику тренировок.
* **AC-1:** *Given* пользователь нажал на челлендж на дашборде, *When* открывается экран деталей, *Then* отображаются: название, общий прогресс-бар (в % и единицах), таймер обратного отсчета до дедлайна, график накопительного прогресса и список всех внесенных тренировок.
* **AC-2:** *Given* список тренировок, *When* пользователь нажимает иконку корзины рядом с тренировкой, *Then* отправляется DELETE запрос на бэкенд, запись удаляется, а график и прогресс-бар пересчитываются на лету.

### US-7: Просмотр списка достижений (Ачивок)
**Как** пользователь, **я хочу** видеть все возможные спортивные достижения в системе, **чтобы** иметь дополнительный стимул их заработать.
* **AC-1:** *Given* пользователь перешел в раздел наград, *When* отображается список ачивок, *Then* каждая ачивка содержит название, описание условий получения, дату разблокировки (если разблокирована) и визуальный статус (активная/заблокированная).

### US-8: Удаление челленджа
**Как** пользователь, **я хочу** удалять ненужный челлендж целиком, **чтобы** держать свой список актуальным.
* **AC-1:** *Given* пользователь находится на экране деталей челленджа, *When* он нажимает «Удалить челлендж» и подтверждает действие в диалоге, *Then* отправляется `DELETE /api/challenges/:id`, челлендж и все его тренировки удаляются, пользователь перенаправляется на дашборд.
* **AC-2:** *Given* попытка удалить несуществующий или чужой челлендж, *When* запрос отправлен, *Then* бэкенд возвращает `404 Not Found`.
* **AC-3:** *Given* интерфейс, *When* пользователь нажимает кнопку «Удалить челлендж», *Then* должен появиться диалог подтверждения (для предотвращения случайного удаления).

### US-9: Бесшовная авторизация через Telegram Mini App
**Как** пользователь, **я хочу** автоматически авторизоваться при запуске приложения в Telegram, **чтобы** мне не нужно было логиниться или вводить ID вручную.
* **AC-1:** *Given* запуск приложения, *When* доступен объект `window.Telegram.WebApp.initData`, *Then* фронтенд использует его в заголовке `Authorization` для всех запросов API.
* **AC-2:** *Given* запрос с валидным `initData`, *When* бэкенд проверяет подпись по токену бота, *Then* извлекается ID пользователя и запрос обрабатывается.
* **AC-3:** *Given* неверный `initData`, *When* запрос отправлен, *Then* возвращается ошибка `401 Unauthorized`.
* **AC-4:** *Given* отсутствие токена бота на бэкенде (режим локальной отладки), *When* запрос отправлен с заголовком `X-User-Id`, *Then* бэкенд пропускает запрос (fallback-режим).

### US-10: Синхронизация визуальной темы
**Как** пользователь, **я хочу**, чтобы приложение выглядело гармонично с моим клиентом Telegram, **чтобы** интерфейс соответствовал светлой/темной теме мессенджера.
* **AC-1:** *Given* загрузка приложения, *When* Telegram передает цвета темы, *Then* все элементы UI (фон, текст, кнопки) окрашиваются с использованием нативных CSS-переменных Telegram (`var(--tg-theme-bg-color)` и т.д.).
* **AC-2:** *Given* смена темы в настройках Telegram, *When* возникает событие `themeChanged`, *Then* интерфейс мгновенно перекрашивается без перезагрузки страницы.

### US-11: Тактильный отклик (Haptic Feedback)
**Как** пользователь, **я хочу** чувствовать вибрацию при ключевых действиях, **чтобы** интерфейс ощущался отзывчивым.
* **AC-1:** *Given* добавление тренировки, *When* API отвечает успехом, *Then* срабатывает легкая вибрация `impactOccurred('medium')` или `notificationOccurred('success')`.
* **AC-2:** *Given* получение ачивки или завершение челленджа, *When* появляется поп-ап, *Then* срабатывает праздничный отклик `notificationOccurred('success')`.
* **AC-3:** *Given* ошибка валидации формы или сбой API, *When* показывается сообщение об ошибке, *Then* срабатывает предупреждающая вибрация `notificationOccurred('error')`.

### US-12: Нативная навигация и закрытие приложения
**Как** пользователь, **я хочу** использовать стандартную кнопку «Назад» в Telegram для переходов и не терять данные при случайном закрытии.
* **AC-1:** *Given* переход на форму создания или детали челленджа, *When* страница открыта, *Then* отображается нативная `BackButton`. Нажатие на нее возвращает на Dashboard.
* **AC-2:** *Given* переход на Dashboard, *When* экран активен, *Then* `BackButton` скрыта.
* **AC-3:** *Given* пользователь начал вводить данные в форму, *When* он пытается закрыть WebApp свайпом вниз, *Then* Telegram показывает нативное предупреждение (`enableClosingConfirmation`), предотвращая потерю данных.

## 6. Development Roadmap (План разработки)

* **Шаг 1: Подготовка инфраструктуры.** Настройка локальной БД / Docker.
* **Шаг 2: Разработка Backend.** Миграции, REST API эндпоинты, логика динамического пересчета статусов `failed` и выдачи ачивок.
* **Шаг 3: Разработка Frontend.** Верстка Dashboard и страниц (Vanilla HTML/CSS/JS). Интеграция с API, обработка поп-апов для ачивок.
* **Шаг 4: Тестирование и отладка.** Проверка граничных условий (прошедшие даты, удаление логов, многократное достижение 100%).
* **Шаг 5: Интеграция Telegram Mini Apps.** Валидация подписи на бэкенде, адаптация стилей, Haptic Feedback и нативные кнопки. (Текущий этап)
